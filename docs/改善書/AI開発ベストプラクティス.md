# AI に改善書・タスクを作成させるベストプラクティス

> NaviOs プロジェクトで実証済みの手法

---

## 1. セッション開始時のプロンプト

### 現状把握を依頼する

```
プロジェクトの現状を把握して、以下をまとめてください：
1. 前回のセッションからの変更点（git log で確認）
2. 未適用のDBマイグレーション
3. 既知のバグ・未実装UI
4. docs/改善書/ のドキュメントを読んで最新状態を確認
```

**ポイント**: AIに「まず読んでから」作業させる。読まずに実装を始めると的外れになる。

---

## 2. 改善案を出させるプロンプト

### 良い例（コンテキスト + 制約 + 出力形式を指定）

```
現在のアプリの状態を精査して、UI/UX の改善案を出してください。

条件：
- 実際のコードを読んで、実装可能な提案のみ
- 各提案に「現状」「問題」「改善案」「工数（S/M/L）」を添える
- 優先度順に並べる
- バグに近い問題は最優先

出力先: docs/改善書/UI_UX改善案_YYYY-MM-DD.md
```

### 悪い例

```
アプリを改善して
```
→ スコープが広すぎて、AIが勝手に判断して不要な変更をする

---

## 3. タスクリストを作成させるプロンプト

### 新規作成

```
docs/改善書/ と docs/案-引継ぎ書・改善書/ の内容を読んで、
現在の残タスクを整理してください。

形式：
- フェーズ別（バグ修正 → 機能追加 → 最適化 → リリース準備）
- 各タスクに「工数」「状態（完了/未着手/一部実装）」「詳細」
- 着手推奨順を理由付きで

出力先: docs/改善書/次回タスク.md
```

### 更新

```
今日のセッションで実装した内容を反映して、
docs/改善書/ のドキュメントをアップデートしてください。
完了したタスクは「完了」に、新しく見つかった問題は追加で。
```

---

## 4. バグ修正を依頼するプロンプト

### テンプレート

```
## バグ報告

### 症状
[具体的に何が起きるか]

### 再現手順
1. ○○画面を開く
2. ○○をタップ
3. → [期待する動作] だが [実際の動作]

### エラーメッセージ（あれば全文貼る）
```
[エラーメッセージ]
```

### プラットフォーム
- [ ] iOS のみ / [ ] Android のみ / [ ] 両方
```

**ポイント**: エラーメッセージは**全文**貼る。コンポーネント名（`Check the render method of 'Xxx'`）があれば特定が一瞬で終わる。

---

## 5. 実装を依頼するプロンプト

### 小さいタスク（1ファイル〜3ファイル）

```
[タスク名]の実装をお願いします。
```

これだけで十分。AIはコードを読んで判断する。

### 大きいタスク（新機能、5ファイル以上）

```
[機能名]の実装を行いたい。
まず設計プランを立ててから実装してください。
```

→ AIが `EnterPlanMode` でプランを作成 → 承認後に実装。手戻りが減る。

### 複数タスクの一括依頼

```
1. [タスクA]
2. [タスクB]
の実装をお願いします。
```

→ 依存関係があればAIが順番を判断する。独立なら並行実装。

---

## 6. ドキュメント更新を依頼するプロンプト

### セッション終了時

```
docs/改善書/ のフォルダを確認してアップデートしてください
```

→ AIが全ドキュメントを読み、今日の実装内容を反映して更新する。

### 特定ドキュメントの更新

```
[ファイルパス] を読んで、現状に合わせて修正してください
```

---

## 7. プロダクト分析を依頼するプロンプト

```
## あなたの役割
プロダクトマネージャーの視点を持つアプリ改善の専門家です。

## アプリの概要
- アプリ名: NaviOs
- ターゲット: 地域住民（20〜50代）、地元店舗・事業者
- 主な機能: 位置情報投稿、タイムライン、検索、ゴースト投稿

## ユーザーフィードバック
---
[フィードバックを貼り付け]
---

## 依頼内容
1. フィードバックの分類と傾向
2. 追加機能の提案（優先度 + 実装難易度付き）
3. 既存機能の改善案
4. 最初に着手すべき1〜2項目とその理由

## 制約
- 個人開発のためスモールスタート優先
- iOS/Android 両対応必須
- Supabase 無料枠内運用
```

---

## 8. やってはいけないこと

| NG | 理由 | 代わりにやること |
|----|------|----------------|
| 「全部直して」 | スコープ不明で品質が下がる | 番号付きリストで個別に列挙 |
| 「UIを良くして」 | 主観的で基準がない | 具体的な画面名 + 期待する状態を書く |
| エラーメッセージを省略 | 原因特定に何倍も時間がかかる | 全文コピペ |
| 1メッセージに10個以上の依頼 | 後半が雑になる | 5個以下に分割 |
| 実装結果を確認せずに次の依頼 | バグが積み重なる | iOS/Android で動確してから次へ |

---

## 9. セッションの理想的な流れ

```
1. 現状把握を依頼（docs読み + git log）
     ↓
2. 前回の残タスク確認（docs/改善書/次回タスク.md）
     ↓
3. 今日やるタスクを選択（ユーザーが決める）
     ↓
4. 実装依頼（小→大の順、1〜3個ずつ）
     ↓
5. 実機確認（iOS + Android）
     ↓
6. バグがあれば報告（エラー全文付き）
     ↓
7. セッション終了時にドキュメント更新を依頼
```

---

## 10. ドキュメント構成ルール

```
docs/改善書/
  ├── 現状分析_YYYY-MM-DD.md    ... セッション時点のスナップショット
  ├── 次回タスク.md              ... 常に最新を維持（上書き更新）
  ├── AI開発ベストプラクティス.md ... 本ファイル（更新頻度低）
  └── UI_UX改善案_YYYY-MM-DD.md  ... 必要に応じて作成
```

- `次回タスク.md` は**毎セッション末に更新**（AIに依頼）
- `現状分析` は**大きな機能追加後**に新規作成（日付をファイル名に）
- 古い現状分析は削除せず残す（変遷を追える）
