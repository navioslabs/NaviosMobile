# ロールバック手順

## EAS Update によるOTAロールバック

EAS Update を使用してJavaScriptバンドルのみのロールバックを行う手順。
ネイティブコード変更を含まないリリースに有効。

### 1. 現在のアップデート状況を確認

```bash
eas update:list --branch production
```

出力例:
```
ID         Group                  Message         Platform  Created
abc123     group-001              v1.2.0 release  ios       2026-03-28
def456     group-000              v1.1.0 release  ios       2026-03-25
```

### 2. ロールバック先のアップデートIDを特定

正常動作していたバージョンのグループIDを確認する。

### 3. ロールバックを実行

```bash
# 特定のアップデートにロールバック（再公開）
eas update:republish --group <group-id>
```

### 4. ロールバック確認

```bash
# 最新のアップデートがロールバック先になっていることを確認
eas update:list --branch production --limit 3
```

ユーザーのアプリは次回起動時にロールバック版を自動取得する。

## EAS Build によるネイティブロールバック

ネイティブコード変更を含むリリースをロールバックする場合。

### 1. 過去のビルドを確認

```bash
eas build:list --platform all --status finished
```

### 2. 過去のビルドを再提出

```bash
# 特定のビルドをストアに再提出
eas submit --platform ios --id <build-id>
eas submit --platform android --id <build-id>
```

### 3. ストア審査

- **iOS**: App Store Connect で審査を待つ（緊急リリースの場合は Expedited Review を申請）
- **Android**: Google Play Console で即座に公開可能（段階的ロールアウト推奨）

## ロールバック判断基準

| 深刻度 | 症状 | 対応 |
| ------ | ---- | ---- |
| Critical | アプリクラッシュ・データ損失 | 即座に EAS Update ロールバック |
| Major | 主要機能が動作しない | 修正パッチを優先、1h以内に修正不可ならロールバック |
| Minor | UI崩れ・軽微な不具合 | 修正パッチで対応（ロールバック不要） |

## 連絡フロー

1. 障害検知（Sentry アラート or ユーザー報告）
2. 影響範囲を評価
3. ロールバック判断（上記基準）
4. ロールバック実行
5. 原因調査・修正
6. 修正版リリース

## 注意事項

- EAS Update はJSバンドルのみ更新するため、ネイティブモジュールの変更を含むバグには対応できない
- ロールバック後もSupabaseのDBマイグレーションは戻らない — DBスキーマ変更を伴うリリースは特に注意
- `eas.json` の `production` プロファイルで `autoIncrement: true` が有効なため、ビルド番号は自動管理される
