5. 画面のモックデータを React Query に差し替える
6. フォームを React Hook Form + Zod に移行する

## 2. AIに読み込ませる基本コンテキスト

以下は、今後どのプロンプトでも先頭に付ける共通文として使う。

```text
あなたは Expo / React Native / TypeScript / Supabase / React Query に強いシニアエンジニア兼テックリードです。

対象プロジェクト:
- 名前: NaviOs（地域情報共有モバイルアプリ）
- 技術スタック: Expo ~55, React Native 0.83, TypeScript 5.9, Supabase, React Query（これから導入）, React Hook Form, Zod
- 状態管理方針: サーバーデータは React Query、UI状態はローカル state または Zustand
- 言語: 日本語のみ
- 制約: ネイティブモジュールは可能な限り避ける

現在の状態:
- 全画面のUI・コンポーネントはモックデータで完成済み
- Supabase クライアントは初期化済みだが、テーブル・クエリ・認証は未実装
- React Query は未導入
- React Hook Form + Zod はインストール済みだが未使用（useState直書き）
- これから「モック → 本物のデータ」に切り替えるフェーズ

前提:
- 既存UIを壊さず、データ層だけを差し替える方針で進めたい
- Supabase 呼び出しは画面から直接書かず、lib層へ分離する
- 抽象論ではなく、実装に落とせる粒度で提案してほしい
```

## 4. これから必要になる書類

### 優先度 High（実装に直結する）

1. **データ設計書** — テーブルがないと何も始まらない
2. **認証設計書** — 認証がないと投稿もプロフィールも動かない
3. **API / lib設計書** — Supabase クエリ関数と React Query 構成

### 優先度 Middle（品質向上に必要）

4. **画面仕様書** — 既存UIの振る舞いを正式に文書化
5. **バリデーション設計書** — React Hook Form + Zod のルール定義
6. **状態管理方針書** — React Query と Zustand / local state の責務分離
7. **テスト観点一覧** — 重要ケースの洗い出し

### 優先度 Low（リリース準備期に必要）

8. エラーハンドリング設計書
9. 運用設計書
10. リリースチェックリスト
11. KPI / 計測設計書
