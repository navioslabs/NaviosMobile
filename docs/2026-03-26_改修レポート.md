## 3. フォーム移行（React Hook Form + Zod）

### 対象画面

| 画面 | ファイル | 移行前 | 移行後 |
|------|---------|--------|--------|
| 投稿作成 | `app/post.tsx` | useState 直書き | useForm + zodResolver(createPostSchema) |
| トーク投稿 | `app/talk-post.tsx` | useState 直書き | useForm + zodResolver(createTalkSchema) |
| プロフィール編集 | `app/profile/edit.tsx` | useState 直書き | useForm + zodResolver(updateProfileSchema) |

### 主な変更点

- **バリデーション**: Zodスキーマによるリアルタイムバリデーション（`mode: "onChange"`）
- **エラー表示**: フィールド単位でバリデーションエラーメッセージを日本語で表示
- **Controller**: TextInput を `Controller` で React Hook Form に接続
- **型安全**: フォーム値が `CreatePostForm` / `CreateTalkForm` / `UpdateProfileForm` 型で保証される

### Zodスキーマの更新

`lib/validations.ts` の `createPostSchema` に `deadline` フィールドを追加:

```ts
export const createPostSchema = z.object({
  category: z.enum(["lifeline", "event", "help"]),
  title: z.string().min(1, "タイトルを入力してください").max(100, "100文字以内で入力してください"),
  content: z.string().max(2000, "2000文字以内で入力してください").optional().or(z.literal("")),
  deadline: z.date({ error: "期限を選択してください" }),
});
```

### プロフィール編集のトグル改善

PrivacyToggle コンポーネントを Controller で接続し、フォーム値として管理:
- `isPublic`, `showLocation`, `showCheckins` がフォーム値として型安全に管理される
- 保存時に全トグル値がまとめて送信される

## 4. エラーハンドリング統一

### 方針

全画面の `catch` ブロックで `getUserMessage(e)` を使用し、ユーザー向け日本語エラーメッセージを統一表示する。

### 変更ファイル

| ファイル | 変更内容 |
|---------|---------|
| `app/post.tsx` | `catch (e: any)` → `catch (e: unknown)` + `getUserMessage(e)` |
| `app/talk-post.tsx` | 同上 |
| `app/profile/edit.tsx` | 同上 |
| `app/(auth)/login.tsx` | 手書きメッセージ分岐 → `getUserMessage(e)` |
| `app/(auth)/signup.tsx` | 手書きメッセージ分岐 → `getUserMessage(e)` |
| `app/talk-detail/[id].tsx` | `e.message` → `getUserMessage(e)` + import追加 |
| `components/ui/ReportModal.tsx` | `e.message` → `getUserMessage(e)` + import追加 |

### エラーメッセージの対応表（`lib/appError.ts`）

| エラーコード | ユーザー向けメッセージ |
|-------------|---------------------|
| `23505` | 既に登録されています |
| `42501` | この操作を行う権限がありません |
| `invalid_credentials` | メールアドレスまたはパスワードが正しくありません |
| `user_already_exists` | このメールアドレスは既に登録されています |
| ネットワークエラー | ネットワークに接続できません |
| JWT/トークンエラー | セッションが切れました。再ログインしてください |
| その他 | エラーが発生しました。もう一度お試しください |

## 5. RLSポリシーレビュー

### レビュー結果

RLSポリシーは適切に設定されている。主要な確認結果:

| テーブル | SELECT | INSERT | UPDATE | DELETE | 評価 |
|---------|--------|--------|--------|--------|------|
| profiles | 公開 or 自分 | — | 自分のみ | — | OK |
| posts | 全員 | 自分のみ | 自分のみ | 自分のみ | OK |
| talks | 全員 | 自分のみ | 自分のみ | 自分のみ | OK |
| comments | 全員 | 自分のみ | 自分のみ | 自分のみ | OK |
| talk_replies | 全員 | 自分のみ | 自分のみ | 自分のみ | OK |
| likes | 全員 | 自分のみ | — | 自分のみ | OK |
| reports | 自分のみ | 自分のみ | — | — | OK |
| Storage | 全員閲覧 | 認証済み | 自分のみ | 自分のみ | OK |

### 注意事項

- `anon`（未認証）ユーザーは投稿・トーク・コメントの **閲覧のみ** 可能（意図通り）
- 他人のデータの編集・削除は `auth.uid() = author_id` で防止されている
- Storage はフォルダ名ベースで所有者チェック（`storage.foldername(name)[1]`）
- `reports` テーブルは自分の通報のみ閲覧可能（他人の通報は見えない）
