# Navios 投稿画面 UI/UX 改善案

> 対象画面: `app/post.tsx`
> 更新日: 2026-03-25
> 位置づけ: 投稿体験の入力負荷軽減と完了率改善のための提案書

---

## 1. 結論

現状の投稿画面は、最低限の投稿機能としては成立している。
ただし、入力項目が縦に並んでいるだけに近く、初見ユーザーにとっては「何をどこまで書けばよいか」が少し見えにくい。

改善方針は以下とする。

- 投稿の流れを `カテゴリを選ぶ -> 何があるかを書く -> 必要なら補足する -> 写真/場所を添える` の順で明確化する
- 必須項目と任意項目の意味を見た目で区別する
- 投稿前の不安を減らし、投稿後の達成感を少し強める
- 情報量を増やしすぎず、1画面で完結する軽さは維持する

---

## 2. 現状の良い点

- カテゴリ選択が先頭にあり、投稿の種類を最初に決められる
- タイトル必須、詳細任意という構成は理解しやすい
- 画像添付、位置情報付与がシンプル
- 投稿前の破棄確認があり、事故が起きにくい
- 送信ボタンはグラデーションで主行動として認識しやすい
- **カテゴリ選択 UI は4分割ボタンにアイコン + ラベルで直感的**（実装確認済み）
- **`AnimatedSubmitButton` で投稿ボタンにタップ感がある**（Reanimated の `withSpring` スケール）
- **`isPending` 時に `ActivityIndicator` を表示済み** — ローディング対応は完了している
- **`canSubmit` が false 時に「タイトルを入力すると投稿できます」の案内が既にある**

---

## 3. 現状の課題

### 3.1 画面の目的が一目で伝わりきらない

カテゴリ、タイトル、詳細、写真、場所が並んでいるが、ユーザーが最初に読むべき説明が弱い。
そのため「この画面で何を投稿するのか」「どの程度書けばよいのか」がやや曖昧に見える。

### 3.2 カテゴリ選択の意味は見えるが、判断支援がもう少し欲しい

カテゴリの説明文はあるが短く、選んだ結果としてどのような投稿に向くのかがまだ弱い。
特に `物資 / イベント / 近助 / 行政` の違いは初回利用時に少し迷いやすい。

> **実装確認**: `CAT_HINTS` で選択中カテゴリの説明が表示される（例: `食料品、日用品など物資の共有に`）。方向性は良いが、もう一段具体的な投稿例があると判断しやすい。

### 3.3 入力途中の安心感が弱い

- タイトルに何を書くべきかの例が弱い
- 詳細欄に「何を書くと良いか」のガイドがない
- 写真と位置情報の価値が伝わりきっていない

> **実装確認**: タイトルのプレースホルダーは `タイトルを入力`、詳細は `詳細を入力` で、どちらも汎用的すぎる。具体例に変えるだけで効果が出る。

### 3.4 投稿完了までの気持ちよさが足りない

現状は投稿完了時にアラートで終了するため、体験としては事務的に見えやすい。
Navios のブランドトーンに合わせるなら、もう少し軽く前向きな完了演出が欲しい。

> **実装確認**: `Alert.alert("投稿完了", "投稿が作成されました")` で OK を押してから `router.back()`。2ステップ必要で、テンポが悪い。

### 3.5 位置情報の文言が機能的すぎる

> **実装確認**: 許可済みの場合 `📍 現在地を取得済み` と表示。事実の伝達としては正しいが、投稿にとっての価値（近くの人に届く）が見えない。

## 4. UI 改善案

### 4.1 上部に「投稿の意味」を示す導入ヘッダーを追加する

先頭に小さな案内領域を置く。

表示案:

- タイトル: `近くの人に、今必要な情報を届ける`
- 補助文: `物資、イベント、助け合い、行政情報を投稿できます`

意図:

- 画面の目的を最初に伝える
- 単なるフォームではなく、地域共有の導線として見せる

> **実装メモ**: `ScrollView` 先頭、カテゴリの前に `Text` 2行追加。`fs.lg` + `WEIGHT.bold` + `t.text` と `fs.xs` + `t.sub`。コスト極小。

### 4.2 カテゴリ選択を「選択中が明確なカード」に寄せる

現状の4分割ボタンの方向性はよいので、以下を追加する。

- 選択中カテゴリだけ説明カードを少し大きく見せる
- カテゴリごとに1行の具体例を出す
- 選択時に上部へ `このカテゴリで投稿します` の小さな確認ラベルを出す

具体例:

- 物資: `野菜入荷 / パンの在庫 / おすそ分け`
- イベント: `地域のお祭り / 体験会 / 集まり`
- 近助: `手伝ってほしい / 手伝えます`
- 行政: `手続き / お知らせ / 公的案内`

> **実装所感**: `CAT_HINTS` を拡張して2行目に具体例を追加するのが最もシンプル。`選択中カードの拡大` は4つのボタンが `flex: 1` の等分レイアウトなので、特定のものだけ大きくするとレイアウトが崩れるリスクがある。**具体例の追加だけに留める方が安全**。

### 4.3 タイトル欄を「最重要入力」として強調する

タイトルは必須であり、一覧で最も目に入る情報でもある。
そのため、通常入力欄より一段強く見せる。

提案:

- プレースホルダーをより具体的にする
- 入力欄の上に短い補助文を入れる
- 入力中はボーダーと背景の反応を少し強くする

文言例:

- ラベル補助: `ひと目で内容が伝わる短い見出しにする`
- プレースホルダー: `例: 越谷駅前で野菜販売 / 今日16時から体操教室`

> **実装メモ**: プレースホルダーの変更は1行。補助文は `Text` 1つ追加。入力中のボーダー反応は `onFocus` / `onBlur` で state を持つ必要があるが、必須ではない。**プレースホルダー + 補助文だけで十分効果が出る**。

### 4.4 詳細欄は「任意だが書く価値がある」ことを見せる

今は任意とだけ見えるため、入力しなくてもよい印象が強い。
任意であることは維持しつつ、書くと便利になる要素を示す。

提案:

- 詳細欄の直下に補助チップを置く
- 例: `時間`, `場所`, `持ち物`, `対象者`, `注意点`
- チップタップで詳細欄に下書きを挿入する方式も有効

意図:

- 何を書けばよいかの迷いを減らす
- 投稿品質を自然に上げる

> **実装所感**: チップタップで下書き挿入は UX として良いが、カーソル位置の管理やテキスト結合のロジックが少し複雑になる。**最小実装はプレースホルダーを `例: 日時、場所、持ち物など` に変えるだけ**。チップ UI は Middle 優先度で後から追加する方が安全。

### 4.5 写真エリアを「追加操作」ではなく「情報補強」として見せる

現状はカメラと画像選択のボタンが並ぶだけなので、少し機械的に見える。
写真の価値を言葉で伝えると添付率が上がりやすい。

提案:

- セクション見出しを `写真を添えると伝わりやすくなります` に近いトーンへ
- 未添付時は大きめの1枚エリアで、`撮影` と `ライブラリから選択` をまとめる
- 添付後はプレビューをより大きく見せる

> **実装確認**: 添付後のプレビューは `height: 200` で十分大きい。未添付時のボタンは `72x72` の dashed border で控えめ。**見出しの文言変更だけで効果が出る**。ボタンサイズの変更は既存レイアウトを壊すリスクがあるため非推奨。

### 4.6 位置情報は「公開される情報」であることを安心して理解できる形にする

今の `現在地を取得済み` は状態表示としては十分だが、ユーザーが気にするのは「どこまで表示されるか」である。

提案:

- 文言を `位置情報つきで近くの人に届きやすくなります` に寄せる
- 許可済みの場合は補助文で `詳細な住所は表示しない` などの安心情報を出す
- 未許可時は `今は位置なしでも投稿可能` か `許可すると近くに届きやすい` のどちらかを明示する

> **実装メモ**: 現在の位置情報カードは `s.card` + `flexDirection: "row"` で、`MapPin` + テキスト + `Locate` アイコンの構成。テキストの差し替えだけで対応可能。

### 4.7 投稿ボタンの上に「送信前サマリー」を出す

投稿直前に、入力内容が不足していないか軽く確認できる要素を置く。

例:

- `カテゴリ: 物資`
- `タイトル: 入力済み`
- `写真: 未設定`
- `位置情報: 取得済み`

これはカード1枚で十分であり、チェックリスト風にすると安心感が増す。

> **実装所感**: 悪くないが、投稿画面は1画面完結の軽さが売り。サマリーを追加すると画面が長くなり、送信ボタンが画面外に押し出される端末が出る。**やるなら、送信ボタン無効時に「不足項目」だけを1行で表示する方が軽い**。現状すでに「タイトルを入力すると投稿できます」の案内があるので、ここを拡張する形がベター。例: `写真を添えるとさらに伝わります`。

### 4.8 完了フィードバックをアラート中心から軽い完了体験へ寄せる

提案:

- 投稿完了時は軽い成功表示を出してから戻る
- 文言は `投稿しました` より `近くの人へ届けました` の方が Navios らしい
- 成功時に微弱なハプティクスを入れる

> **実装メモ**: 現在は `Alert.alert("投稿完了", ...)` で OK を押してから `router.back()`。改善案は2つ:
>
> **案A（最小）**: Alert を削除し、ハプティクス + `router.back()` のみ。トーク画面と同じ方式。シンプルだが「投稿できた」の確信が弱い。
>
> **案B（推奨）**: Alert を削除し、画面内に一瞬の成功表示（`近くの人へ届けました`）を出してから 800ms 後に `router.back()`。`Animated.View` + `FadeIn` で表示。ハプティクスも併用。テンポがよく、Navios のトーンに合う。
>
> 案B を推奨する。実装コストは state 1つ + Animated.View 1つ + setTimeout で 20行程度。

## 5. UX 改善案

### 5.1 初回ユーザー向けの迷いを減らす

- 各入力欄に「例」を入れる
- カテゴリ選択後に、適した投稿例を1つ見せる
- 詳細は任意でも、書くと役立つ内容を示す

### 5.2 投稿失敗時の再挑戦をやさしくする

アラートだけで終えるより、画面内のエラー表示があると再挑戦しやすい。

提案:

- 送信ボタン下にエラーバナーを表示
- 画像アップロード失敗と投稿失敗を分けて表現
- 入力内容は失敗後も保持する

> **実装確認**: 入力内容の保持は現状で動作する（Alert のみで画面遷移しないため）。エラーバナーは `createPostMutation.error` を参照すれば実装できるが、現状の Alert でも大きな問題はない。**Middle 優先度で十分**。

### 5.3 長いフォームに見せない

実際の項目数は多くないが、見え方としてはやや縦長である。
セクションを「必須」「あると便利」に分けると心理的負荷が下がる。

構成案:

- 必須: カテゴリ、タイトル
- あると便利: 詳細、写真、位置情報

> **実装所感**: 現状すでに「任意」ラベルで区別されている。セクション分けの見出し（例: `--- あると便利 ---`）を追加するとかえって画面が長くなる。**現状の「任意」ラベルの位置と見た目を少し強調する程度で十分**。

### 5.4 投稿の価値をユーザーに返す

投稿前に小さく次の文言を入れると、行為の意味が強くなる。

- `今の情報が、近くの誰かの判断を助けます`

> **実装メモ**: 導入ヘッダー（4.1）に含めれば1箇所で済む。独立した要素として追加すると画面が冗長になるため、ヘッダーの補助文として統合を推奨。

## 6. 優先度

### High（実装コスト低・効果高）

1. タイトル・詳細のプレースホルダーを具体例に変更
2. `CAT_HINTS` に投稿例を追加
3. 位置情報の文言を価値訴求型に変更
4. 投稿完了を Alert から画面内成功表示 + ハプティクスに変更

### Middle（効果高だが変更範囲がやや大きい）

5. 導入ヘッダーの追加
6. 写真セクションの見出し文言改善
7. 送信前の不足項目ガイド拡張（ボタン下テキストの強化）
8. 投稿失敗時の画面内エラーバナー

### Low（やるなら丁寧にやる）

9. 詳細欄の下書き支援チップ
10. カテゴリ選択カードの拡大演出

> **優先度変更の理由**: 元案の「送信前サマリー」(4.7) は画面が長くなるリスクがあるため Low に降格。代わりに「投稿完了体験の改善」を High に昇格 — Alert 2タップの体験は明確にテンポが悪く、修正の投資対効果が高い。

## 7. 画面構成の推奨イメージ

1. 導入ヘッダー（1〜2行、控えめ）
2. カテゴリ選択（投稿例つき）
3. タイトル（具体例プレースホルダー）
4. 詳細（具体例プレースホルダー）
5. 写真（価値訴求の見出し）
6. 位置情報（価値訴求の文言）
7. ~~投稿前サマリー~~ → 不足時のガイドテキスト（ボタン下に統合）
8. 投稿ボタン（既存 AnimatedSubmitButton を維持）
9. 投稿成功時: 画面内成功表示（FadeIn + ハプティクス → 自動で戻る）

## 9. 備考

この画面は「重い投稿作成フォーム」ではなく、「地域情報を軽く共有するための投稿導線」として完成させるのが望ましい。
そのため、入力補助は強めても、確認ダイアログや手順の分割は増やしすぎない方がよい。

> **追記**: 現在のコードは225行で、`AnimatedSubmitButton` を含む。Reanimated のアニメーション、`useImagePicker` / `useLocation` / `useCreatePost` の hook 連携、破棄確認まで実装済み。改善のベースとして十分な完成度がある。フォーカスすべきは **テキストの改善**（プレースホルダー、文言）と **投稿完了体験の改善**（Alert → 画面内フィードバック）の2点。
