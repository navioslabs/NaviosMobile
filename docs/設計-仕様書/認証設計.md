> 対象: NaviOs
> 更新日: 2026-03-25

## 1. 認証方式

Supabase Auth を使用。MVPではメール+パスワード認証のみ。

| 項目 | 方針 |
|------|------|
| 認証プロバイダ | メール+パスワード（MVP） |
| ソーシャルログイン | 将来対応（Google, Apple） |
| セッション管理 | Supabase Auth 標準（JWT + Refresh Token） |
| セッション永続化 | `@react-native-async-storage/async-storage` |
| ゲスト利用 | 閲覧のみ可能。投稿・いいね・コメントはログイン必須 |

## 2. 認証フロー

### 2.1 サインアップ

```
ユーザー → サインアップ画面 → メール+パスワード入力
  → supabase.auth.signUp()
  → auth.users にレコード作成
  → DB トリガーで profiles テーブルにレコード自動作成
  → セッション発行 → ホーム画面へ遷移
```

### 2.2 ログイン

```
ユーザー → ログイン画面 → メール+パスワード入力
  → supabase.auth.signInWithPassword()
  → セッション発行 → ホーム画面へ遷移
```

### 2.3 ログアウト

```
設定画面 → ログアウトボタン
  → supabase.auth.signOut()
  → セッション破棄 → ログイン画面へ遷移
```

### 2.4 セッション復元（アプリ起動時）

```
アプリ起動 → supabase.auth.getSession()
  → セッションあり → ホーム画面
  → セッションなし → ゲストとしてホーム画面（閲覧モード）
```

## 3. profiles テーブル自動作成トリガー

Supabase Auth で新規ユーザーが作成されたとき、`profiles` テーブルに初期レコードを自動挿入する。

```sql
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS trigger AS $$
BEGIN
  INSERT INTO public.profiles (id, display_name, avatar_url)
  VALUES (
    NEW.id,
    COALESCE(NEW.raw_user_meta_data->>'display_name', 'ユーザー'),
    NULL
  );
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();
```

## 4. useAuth hook 設計

`hooks/useAuth.ts` に実装する。

```typescript
interface AuthState {
  user: User | null;        // auth.users のデータ
  profile: Profile | null;  // profiles テーブルのデータ
  isLoading: boolean;       // 初期セッション復元中
  isGuest: boolean;         // 未ログイン状態
}

interface AuthActions {
  signUp(email: string, password: string, displayName: string): Promise<void>;
  signIn(email: string, password: string): Promise<void>;
  signOut(): Promise<void>;
}
```

**実装方針**:
- `supabase.auth.onAuthStateChange()` でセッション変化を監視
- ログイン成功時に `profiles` テーブルからプロフィール取得
- Context Provider で全アプリに認証状態を配信

## 6. Expo Router での認証リダイレクト

`app/_layout.tsx` でセッション状態に応じてルーティングを制御する。

```
アプリ起動
  ├─ セッション復元中 → スプラッシュ / ローディング表示
  ├─ ログイン済み → (tabs) へ遷移
  └─ 未ログイン → (tabs) へ遷移（ゲストモード）
       └─ 投稿・いいね等の操作時 → (auth)/login へ遷移
```

**ゲストモード方針**:
- ホーム、ちかく、ひとこと、さがす → 閲覧可能
- 投稿作成、いいね、コメント、通報 → タップ時にログイン画面へ誘導
- 設定画面 → プロフィールセクションに「ログインする」ボタン表示

## 7. ゲスト制限の実装方針

### 7.1 AuthGuard コンポーネント

認証が必要な操作をラップするコンポーネント。

```typescript
// 使用例: いいねボタン
<AuthGuard onGuest={() => router.push('/(auth)/login')}>
  <LikeButton onPress={handleLike} />
</AuthGuard>
```

### 7.2 制限対象アクション一覧

| アクション | ゲスト | ログイン済み |
|-----------|--------|-------------|
| フィード閲覧 | 可 | 可 |
| 投稿詳細閲覧 | 可 | 可 |
| ひとこと閲覧 | 可 | 可 |
| ちかく閲覧 | 可 | 可 |
| 検索 | 可 | 可 |
| 投稿作成 | 不可 → ログイン誘導 | 可 |
| ひとこと投稿 | 不可 → ログイン誘導 | 可 |
| いいね | 不可 → ログイン誘導 | 可 |
| コメント | 不可 → ログイン誘導 | 可 |
| 通報 | 不可 → ログイン誘導 | 可 |
| プロフィール編集 | 不可 | 可 |
| テーマ / フォントサイズ変更 | 可 | 可 |

## 8. セキュリティ考慮事項

| 項目 | 対策 |
|------|------|
| パスワード保存 | Supabase Auth が bcrypt でハッシュ化（クライアント関与なし） |
| JWT | Supabase が自動管理、Refresh Token で更新 |
| 環境変数 | `SUPABASE_ANON_KEY` のみクライアント使用。Service Role Key はクライアントに含めない |
| RLS | 全テーブルで有効化。認証状態に基づくアクセス制御 |
| レート制限 | Supabase Auth のデフォルト制限に依存（MVP） |
